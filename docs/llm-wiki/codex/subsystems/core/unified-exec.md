> Unified Exec 是 Codex 的交互式进程执行子系统：tool registry 暴露 `exec_command` 和 `write_stdin`，handler 解析模型参数，process manager 经由 `ToolOrchestrator::run` 进入 approval/sandbox 协议，并把仍存活的进程保存在 session 级 manager 中供后续 stdin/poll 使用。[E: codex-rs/tools/src/tool_registry_plan.rs:157][E: codex-rs/tools/src/tool_registry_plan.rs:165][E: codex-rs/tools/src/tool_registry_plan.rs:169][E: codex-rs/tools/src/tool_registry_plan.rs:170][E: codex-rs/core/src/tools/handlers/unified_exec.rs:202][E: codex-rs/core/src/tools/handlers/unified_exec.rs:371][E: codex-rs/core/src/unified_exec/process_manager.rs:830][E: codex-rs/core/src/unified_exec/process_manager.rs:278]

## 能回答的问题

- `exec_command` / `write_stdin` 的输入字段和默认值是什么？
- 为什么 `exec_command` 支持 parallel tool calls，而 `write_stdin` 不支持？
- Unified exec 如何在初始 yield 前保存 live process？
- PTY、pipe、remote exec-server、ZshFork 路径如何分流？
- approval、sandbox first attempt、sandbox denial retry 如何串联？
- output buffering、delta events、end events 和 tool response 如何对应？
- session shutdown 为什么会 terminate 所有 unified exec processes？

## 设计动机与权衡

Unified Exec 把 policy/user interaction 放在 `ToolOrchestrator`，把 PTY/pipe/remote process 生命周期放在 `unified_exec` 模块；这种分离让 approval/sandbox retry 和 process management 可以独立演进。[E: codex-rs/core/src/tools/orchestrator.rs:124][E: codex-rs/core/src/tools/orchestrator.rs:362][E: codex-rs/core/src/unified_exec/process_manager.rs:664][E: codex-rs/core/src/unified_exec/process_manager.rs:725][E: codex-rs/core/src/unified_exec/process_manager.rs:737][I]

`write_stdin` 不支持 parallel tool calls，是因为它操作已有 process 的 stdin/poll 状态；registry 明确把它的 `supports_parallel_tool_calls` 设为 false，而 `exec_command` 设为 true。[E: codex-rs/tools/src/tool_registry_plan.rs:161][E: codex-rs/tools/src/tool_registry_plan.rs:166][I]

Initial yield 前保存 live process 是为了支持 turn 被中断后后台进程仍可续接；源码在 initial yield wait 之前判断进程仍 alive 并调用 `store_process`。[E: codex-rs/core/src/unified_exec/process_manager.rs:273][E: codex-rs/core/src/unified_exec/process_manager.rs:274][E: codex-rs/core/src/unified_exec/process_manager.rs:278][I]

Head/tail buffer 牺牲中间 output 来保留开头上下文和最新尾部，这与 CLI/terminal 场景中“命令开头 + 最新错误尾部”更有检索价值的经验一致。[E: codex-rs/core/src/unified_exec/head_tail_buffer.rs:72][E: codex-rs/core/src/unified_exec/head_tail_buffer.rs:81][E: codex-rs/core/src/unified_exec/head_tail_buffer.rs:86][E: codex-rs/core/src/unified_exec/head_tail_buffer.rs:90][I]

## gotcha

- `tty=false` 的 process 没有 stdin；非空 `write_stdin` 会返回 `StdinClosed`，空 `write_stdin` 可以作为 poll 使用。[E: codex-rs/core/src/unified_exec/process_manager.rs:428][E: codex-rs/core/src/unified_exec/process_manager.rs:430][E: codex-rs/core/src/unified_exec/process_manager.rs:457]
- `max_output_tokens` 不控制底层 buffer bytes；`HeadTailBuffer` 的 retained bytes 由 `UNIFIED_EXEC_OUTPUT_MAX_BYTES` 控制，tool response 只是携带 `max_output_tokens` 字段。[E: codex-rs/core/src/unified_exec/mod.rs:65][E: codex-rs/core/src/unified_exec/process_manager.rs:397][I]
- `exec_command` 被 apply_patch intercept 时会先 allocate process id，但 intercept 后立即 release，并且 tool output 的 `process_id` 为 None，因此不会保留 live process 供 `write_stdin` 续接。[E: codex-rs/core/src/tools/handlers/unified_exec.rs:211][E: codex-rs/core/src/tools/handlers/unified_exec.rs:294][E: codex-rs/core/src/tools/handlers/unified_exec.rs:306][E: codex-rs/core/src/tools/handlers/unified_exec.rs:307][E: codex-rs/core/src/tools/handlers/unified_exec.rs:313]
- Process pruning 保护最近 8 个 process，优先 prune 已退出且不在 protected set 的 LRU process，否则 prune 不受保护的 LRU process。[E: codex-rs/core/src/unified_exec/process_manager.rs:1003][E: codex-rs/core/src/unified_exec/process_manager.rs:1007][E: codex-rs/core/src/unified_exec/process_manager.rs:1016][E: codex-rs/core/src/unified_exec/process_manager.rs:1022]
